# AI Coding 分享提纲

## 分享主题

从「AI 辅助补全」到「Vibe Coding」：
我如何在 **没有手写一行项目代码、没有手配任何环境** 的前提下，把 `rem-ai` 做到可远程预览、可轻量远程开发。

## 一句话版开场

过去我们把 AI 当成“高级补全”来用；这次我想分享的是另一种工作方式：
**把 AI 从代码助手，升级成项目执行搭档。**

我这次做 `rem-ai`，基本没有自己手写代码，也没有自己手动配环境，但最后项目能跑、能远程预览、能轻量接力开发。

重点不是“人不需要了”，而是：
**人开始从“逐行写代码”转向“定义目标、校正方向、验收结果”。**

## 适合周会的结构

### 1. 我原来怎么用 AI

- 主要停留在编辑器补全、插件问答、局部改代码。
- 本质还是“我主写，AI 辅助”。
- 优点是安全，但放大倍数有限。

### 2. 这次我实际做了什么

- 我选了一个真实项目：`rem-ai`。
- 我没有自己手写项目代码。
- 我也没有自己去配本机环境、远程预览、Cloudflare Tunnel 这些基础设施。
- 我做的是：
  - 讲清目标
  - 持续验收
  - 发现偏差就纠正
  - 决定优先级和取舍

### 3. 什么叫 Vibe Coding

- 不是“让 AI 随便写”。
- 更像是：**你描述意图，AI 负责展开实现，你负责校准。**
- 人的角色从“打字员”变成：
  - 产品定义者
  - 技术判断者
  - 验收者
  - 风险控制者

### 4. 这次 `rem-ai` 里，AI 不只是写业务代码

- 排查服务为什么起不来
- 修开发环境
- 接 Docker 的 Postgres / Redis
- 修 WebSocket 接入问题
- 接 Cloudflare Tunnel
- 做远程预览
- 做轻量远程终端
- 加共享密码门禁
- 写脚本和文档

这说明 AI coding 的价值不只是“写一个函数”，而是：
**参与工程化全过程。**

### 5. 这次项目里我体感最强的能力

- 速度快：问题暴露后，AI 可以立刻沿着日志、配置、代码一路查到底。
- 跨层协同：前端、后端、部署、文档、脚本可以一起推进。
- 上下文连续：前面遇到的问题、后面做的折中、最终方案能被串起来。
- 对个人能力放大很明显：哪怕我自己没手写代码，也能推动项目往前走。

### 6. 但它不是“自动成功”

Vibe coding 也有门槛，难点主要有 4 类：

- **目标不清**
  - 你说不清最终要什么，AI 就会来回试错。
- **边界不清**
  - 不限制写入范围，就容易和别的改动冲突。
- **验证不严**
  - AI 很会“看起来像完成了”，但真正可用还得看运行结果。
- **系统问题比编码问题更难**
  - 真正卡住项目的，经常不是业务逻辑，而是环境、权限、网络、依赖、部署。

### 7. 这次 `rem-ai` 的真实技术难点

- 家里电脑是 `MacBook Air 16GB`，不适合重型全容器方案。
- 公司电脑在办公网，不能用 Tailscale。
- 要实现：
  - 家里电脑跑项目
  - 公司电脑能预览
  - 最好还能远程接力开发
- 遇到的实际问题包括：
  - Docker 常驻太重
  - `nodemon` 缺失
  - `whisper-server` 缺失导致服务启动崩
  - Next 开发态缓存异常
  - WebSocket 在 HTTPS 下的 `ws/wss` 问题
  - 存储层把 `"dev"` 当成 UUID
  - Cloudflare Access 受计划限制，先用本地密码门禁兜住

### 8. 最后落地成了什么方案

- 应用原生跑在家里电脑上
- Docker 只保留 Postgres / Redis
- `Cloudflare Tunnel` 提供远程 HTTPS 预览
- `app-rem.remi.run` 负责页面预览
- `term-rem.remi.run` 负责轻量远程终端
- 不上重型浏览器 IDE，优先 `ttyd + tmux`
- 先用共享密码门禁保护公网入口

### 9. 我想传达的结论

- AI coding 已经不只是补全代码。
- 它可以把一个人的推进能力放大到“跨前后端 + 部署 + 文档 + 运维”。
- 但前提是：
  - 目标要清晰
  - 过程要持续验收
  - 关键风险要人工拍板

### 10. 不制造焦虑的收尾

- 不是要求每个人立刻“零手写代码”。
- 更现实的升级路径是：
  1. 从补全开始
  2. 再到让 AI 负责一个完整小功能
  3. 再到让 AI 帮你推进一个完整工程任务

- 不是“程序员会不会被替代”。
- 而是：
  **会不会有人先学会把 AI 变成工程放大器。**

## 技术难点总结

### 云部署/远程访问难点

- 办公网限制
- 不方便走传统 SSH / Tailscale
- 要求 HTTPS / 浏览器友好
- 本机资源有限，不能上太重的常驻方案

### 项目工程难点

- WebSocket 在公网 HTTPS 下要切 `wss`
- 本地依赖和远程访问路径要统一
- 服务启动顺序和缓存状态容易导致假问题
- 数据库、Redis、TTS/STT 等依赖会放大调试复杂度

### Vibe coding 难点

- 描述不清目标时，AI 会乱飘
- 如果不设边界，容易和别人的改动冲突
- 如果不验收，AI 会“看起来完成”
- 基础设施问题往往比功能代码更难
