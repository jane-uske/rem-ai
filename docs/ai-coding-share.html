AI Coding Share 从 AI 辅助补全，到真正的 Vibe Coding
开场 案例 难点 演示 结论
01 · 开场

我没有手写一行项目代码，也没有手配任何环境，但把 rem-ai 推到了可远程预览、可轻量远程开发。

这次分享不是讲“AI 帮我补全了几段代码”，而是讲一种新的工作方式：把 AI 从代码助手，升级成项目执行搭档。

重点不是“人不需要了”，而是人的角色开始从“逐行写代码”，转向“定义目标、校正方向、验收结果”。
02 · 背景

大多数团队对 AI coding 的理解，还停留在“补全更快一点”。

过去的用法
  • 编辑器补全
  • 插件问答
  • 局部生成函数、单个组件
  • 本质仍是“我主写，AI 辅助”
这次的用法
  • 我给目标和约束
  • AI 推进代码、环境、脚本、部署、文档
  • 我持续验收和纠偏
  • 本质变成“我主导，AI 执行工程链路”
03 · 案例

`rem-ai` 不是 demo，它是一个真的在跑的 AI 陪伴项目。

它有对话、情绪、记忆、语音、WebSocket、前端渲染、数据库和缓存。也就是说，AI coding 这次面对的不是一个简单页面，而是一条完整工程链路。

项目能力

聊天、记忆、情绪、TTS/STT、WebSocket、3D Avatar。

工程挑战

本机资源有限、办公网限制、远程预览、远程开发、访问安全。

我的角色

不给代码细节，而是给目标、限制、验收标准和优先级。

04 · 这次 AI 真正做了什么

不只是写业务代码，而是参与了整个工程闭环。

环境原生运行 + Docker 仅保留 Postgres / Redis
部署Cloudflare Tunnel 接通 HTTPS 远程预览
接入处理 WebSocket、`ws/wss`、公网域名
安全先加本地共享密码门禁，不让公网裸奔
开发做出 `ttyd + tmux` 轻量远程终端
文档补脚本、启动流程、远程开发说明
这说明 AI coding 的价值，不只是“写一个函数”，而是开始接管真实项目里的多层问题。
05 · 技术难点

真正卡住项目的，往往不是功能代码，而是系统问题。

基础设施难点
  • 家里机器是 MacBook Air 16GB，不适合重型全容器方案
  • 公司电脑在办公网，不能走 Tailscale
  • 要能从公司网络直接访问家里电脑上的最新效果
实际踩到的问题
  • `nodemon` 缺失
  • `whisper-server` 缺失导致启动崩
  • Next 开发缓存/HMR 异常
  • HTTPS 下 WebSocket 要切到 `wss`
  • 存储层把 `"dev"` 当 UUID
  • Cloudflare Access 受 plan 限制，先用本地密码门禁兜住
06 · Vibe Coding 的难点

Vibe Coding 不是“让 AI 自由发挥”，它也有门槛。

为什么会失控
  • 目标不清，AI 就会乱试
  • 边界不清，容易和别人的改动冲突
  • 验证不严，AI 很会“看起来完成了”
我的体会
  • 人不再逐行写代码，但必须持续验收
  • 真正的价值来自目标定义与取舍判断
  • 系统问题越复杂，越需要人把问题收窄
Vibe coding 不是“不要工程纪律”，而是把工程纪律从写代码阶段，前移到任务设计和验收阶段。
07 · 现场演示

演示重点不是炫技，而是让同事看到 AI coding 已经能推进真实项目。

演示 1：远程预览

打开 `app-rem.remi.run`，输入共享密码，展示页面和聊天链路是活的。

演示 2：轻量远程开发

打开 `term-rem.remi.run`，进入 `ttyd + tmux`，展示 agent / 命令 / 日志。

演示 3：工作流变化

强调你不是自己一行行写，而是在持续定义目标、修正方向、验收结果。

08 · 最后的落地方案

为了适配办公网和轻量机器，最终不是“云 IDE”，而是更克制的组合。

预览`app-rem.remi.run`
终端`term-rem.remi.run`
运行应用原生跑，Docker 只保留数据库和缓存
安全先用共享密码门禁，后续再上 Cloudflare Access
体验不用重型 IDE，也能实现接力开发
09 · 想传达的结论

AI coding 已经不只是“补全更快一点”，而是在把个人的工程推进能力放大。

  • 它可以跨前端、后端、部署、脚本、文档一起推进。
  • 它特别适合把个人从“逐行实现”中解放出来，转去做目标定义和结果验收。
  • 但它不等于“不要技术判断”，反而更依赖判断、边界感和验证能力。
不是“程序员会不会被替代”，而是：会不会有人先学会把 AI 变成工程放大器。
10 · 不散播焦虑的收尾

更现实的升级路径，不是一下子零手写，而是逐步把 AI 从工具变成搭档。

阶段 1

先用补全、问答、重构建议。

阶段 2

让 AI 负责一个完整小功能，你负责验收。

阶段 3

像这次一样，让 AI 推进一整条工程链路，你做人机协作的总控。

建议分享时间：10-15 分钟。节奏上少讲抽象观点，多讲 `rem-ai` 这个真实案例和你自己的体验变化。
